# FSD 论文“流形/Isomap 改距离”方案总结（给 Cursor 的讨论提示词）

**用途：** 把这份文档直接贴给 Cursor，讨论 FSD 论文中如何把原始欧氏距离改成更接近流形/Isomap 思想的图距离实现。

## 一、我的目标

- 我想在 FSD（Few-Shot AIGI Detector）这篇论文的基础上，尽量少改主干网络和训练流程，只改“距离计算”这部分，让方法从原来的欧氏距离度量，变成更接近流形学习思想的距离度量。
- 我目前最倾向的方案不是上来做复杂的严格流形建模，而是采用一条可落地、逻辑顺、代码改动相对小的路线：KNN 建图 + 最短路径距离（Isomap 风格）替换原始欧氏距离。
- 我希望你结合代码，帮我判断这个方案在当前 FSD 仓库里应该改哪里、怎么改、哪些地方最容易出 bug、训练/测试流程是否都要改。

## 二、我目前对相关概念的理解

- 1. KNN 本身不等于流形学习。KNN 只是一个找邻居/建图的工具。
- 2. 如果只是“找 k 个近邻”然后做投票或平均，这不太能算真正用了流形方法。
- 3. 如果是“KNN 建图 + 图上的最短路径距离近似测地距离”，这就更接近 Isomap/流形学习思想。
- 4. 所以在论文表述上，更稳妥的说法是：引入基于 Isomap 的流形距离建模，或引入流形学习启发的距离度量，而不是夸大成完整严格的流形建模。

## 三、为什么想改 FSD

- FSD 原始做法里，query feature 和各个 class prototype 之间通常是直接算欧氏距离，再送进 softmax 做分类。
- 这默认特征空间是“平的”。但我怀疑真实特征分布可能是非线性的、弯曲的，只用欧氏距离可能不能很好反映 query 和 prototype 的真实关系。
- 因此我想尝试：不再只看 query 到 prototype 的直线距离，而是看它们在局部邻域图结构中的路径距离。

## 四、我想采用的核心方案（最重要）

- 请优先围绕下面这个版本来讨论代码改法，不要一开始就给我太复杂、太重构的方案。
- 方案主线：保持 backbone、feature extractor、episode 构造方式、prototype 求法基本不变，只替换“query 到 prototype 的距离计算”。
- 原始形式：d(z_q, c_k) = || z_q - c_k ||_2
- 目标形式：d_graph(z_q, c_k) = shortest_path(z_q, c_k)
- 其中 shortest_path 不是直接在原图像空间算，而是在当前 episode 的特征图上计算：把 support feature、query feature、prototype 一起看成图中的节点；先基于特征欧氏距离找 KNN 邻居并建图；边权为局部欧氏距离；再计算 query 到每个 prototype 的图最短路径距离；最后把这个图距离代替原来的欧氏距离送入分类。

## 六、我更偏好的代码改法顺序

- 第一步：不要改 backbone，不要改数据读取，不要改原型求法，先把原有欧氏距离那条链路定位清楚。
- 第二步：先做一个最小可运行版本：在当前 episode 内，对 support/query/prototype 特征建图，用最短路径距离替换欧氏距离。
- 第三步：先只保证能 forward、能 train、能 eval，不要求一上来就最优。
- 第四步：如果这个最小版本跑通，再讨论是否加入更稳健的图构建策略，比如对边权做高斯核、对图做对称化、对 prototype 的接入方式做调整。

## 七、我希望的伪代码（让 Cursor 对照代码实现）

- 输入：support features, query features
- 步骤 1：根据 support features 求每个类的 prototype c_k
- 步骤 2：把当前 episode 中要参与图计算的点整理成一个节点集合 V = {support, query, prototypes}
- 步骤 3：计算这些节点之间的局部欧氏距离矩阵
- 步骤 4：对每个节点找 k 个最近邻，构造 KNN 图
- 步骤 5：边权使用局部欧氏距离（也可以讨论是否改成 exp(-d^2 / sigma^2)）
- 步骤 6：对每个 query，计算其到各 prototype 的最短路径距离 d_graph
- 步骤 7：把 d_graph 作为新的类别距离，送入原有 softmax / classification loss
- 步骤 8：检查 backward 是否可行；如果最短路部分不可导，先确认当前实现是否默认只在 episode 推理层使用，或是否需要近似/替代方案
