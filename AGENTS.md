## 3. 分支规则

本项目目前主要有两个分支：

- main：论文原始 baseline 版本，用于保存干净的原始代码，不要在这个分支上做实验修改。
- exp-isomap-distance-v1：当前实验分支，用于实现和测试 Isomap / kNN / 图距离相关修改。

后续可能会增加新的实验分支。

新增实验分支时，应遵守以下规则：

- main 永远保留论文原始 baseline，不直接改实验代码。
- 每一种较大的实验方向都应该单独建一个 exp-* 分支。
- 不同实验方向不要混在同一个分支里。
- 如果只是修复当前实验分支的小 bug，可以继续在当前分支修改。
- 如果是新方案、新模块、新消融实验，优先新建分支。

推荐分支命名方式：

- exp-isomap-distance-v1：流形距离 / 图距离 / Isomap 距离实验。
- exp-frequency-branch-v1：频域分支实验。
- exp-dual-branch-v1：像素域 + 频域双分支融合实验。
- exp-isomap-dual-branch-v1：双分支 + 图距离组合实验。
- exp-ablation-v1：消融实验分支。
- fix-xxx：小修复分支。
- docs-xxx：文档整理分支。

除非用户明确说明，否则当前默认代码修改分支是：

- exp-isomap-distance-v1

修改代码前必须确认当前分支：

- git branch

如果不在目标实验分支，应先切换。

例如切换到当前实验分支：

- git switch exp-isomap-distance-v1

不要在 main 分支上直接改实验代码。

如果用户提出一个新的实验方向，例如新增频域分支、双分支融合、替换模型结构、增加新的距离度量，请先提醒用户是否需要新建分支，不要直接在当前分支里混合实现。

## 4. 当前方案记忆

当前已经讨论过的主要实验方向包括：

1. Isomap / 图距离方向

这个方向是在 FSD 原始 prototype 分类流程中修改距离计算方式。

原始 FSD 通常是让 query 特征和每个 prototype 计算欧氏距离。

当前实验想法是：

- 先保留原来的欧氏距离作为 baseline。
- 再新增 kNN / 图距离 / Isomap 风格距离。
- 用图上的最短路径距离，部分替代原来的直线距离。
- 重点是观察这种距离是否更适合弯曲分布的特征空间。

解释时请注意：

- 不要只说“流形学习”。
- 要说清楚它在代码里具体改的是距离计算部分。
- 要说清楚训练、验证、测试中哪些地方会用到这个距离。
- 要保留欧氏距离开关，方便做对比实验。

2. 频域分支方向

这个方向是后续可能新增的实验，不要默认已经完成。

大致想法是：

- 原始 FSD 主要从图片本身提取像素域特征。
- 新方案可能额外增加一个频域分支。
- 频域分支可以通过傅里叶变换或小波变换，把图片转换成频域表示。
- 频域表示再送入特征提取器，得到频域特征向量。
- 最后把像素域特征和频域特征融合，再进入 prototype / distance / classification 流程。

解释时请注意：

- 频域分支不是简单替换原始 FSD，而是作为补充分支。
- 原始像素域分支应尽量保留。
- 新增频域分支时应提供开关，方便只用原始方法、只用频域方法、或使用双分支方法。
- 如果两个分支都使用 ResNet，需要解释它们输入信息不同：一个看原图或像素特征，一个看频域表示，因此不是简单重复。
- 不要把该方案和 Co-Spy 混淆。Co-Spy 更偏语义分支和伪影分支；当前计划的方案更偏像素域规律与频域规律的互补。

3. 双分支融合方向

后续可能把像素域特征和频域特征融合。

可能的融合方式包括：

- 直接拼接两个特征向量。
- 加权相加两个特征向量。
- 用一个小的融合层学习权重。
- 先分别计算两个分支的距离，再融合距离分数。

实现时请注意：

- 优先选择工程上简单、容易消融、容易解释的融合方式。
- 不要一开始就做复杂注意力融合。
- 要能分别跑原始 FSD、频域单分支、双分支融合，方便做消融实验。
- 每种融合方式最好用参数开关控制。

## 5. Git 同步流程

远程仓库关系：

- origin 指向 GitHub 仓库：https://github.com/zenghao0718/MC-MPD.git
- GitHub 是 Windows 本地与 AutoDL 之间的主要同步中转站。
- AutoDL 上旧的 autodl-local 远程只作为历史中转仓库保留，一般不要再使用。

标准流程是：

1. Windows 本地改代码
2. 本地 commit 保存版本
3. push 到 GitHub
4. AutoDL pull GitHub 最新代码
5. AutoDL 跑实验

Windows 本地项目路径通常是：

- D:\FSD_work\Few-Shot-AIGI-Detector-main

Windows 本地常用命令：

- git switch 目标分支名
- git status
- git add .
- git commit -m "简要说明本次修改"
- git push

命令含义：

- git switch 目标分支名：切换到要修改的分支。
- git status：查看哪些文件被修改。
- git add .：把修改加入暂存区。
- git commit：在本地保存一个版本。
- git push：把本地提交上传到 GitHub。

AutoDL 项目路径通常是：

- /root/autodl-tmp/Few-Shot-AIGI-Detector-main

AutoDL 常用命令：

- cd /root/autodl-tmp/Few-Shot-AIGI-Detector-main
- git switch 目标分支名
- source /etc/network_turbo
- git pull --ff-only
- git status

命令含义：

- cd：进入 AutoDL 上的项目目录。
- git switch 目标分支名：切换到要运行实验的分支。
- source /etc/network_turbo：开启 AutoDL 学术加速，避免 GitHub 连接超时。
- git pull --ff-only：从 GitHub 拉取最新代码，只允许快进式更新，避免自动产生合并提交。
- git status：确认 AutoDL 工作区是否干净。

## 7. FSD 项目重点

解释或修改 FSD 代码时，请特别关注：

- 原始 FSD 方法在哪里实现。
- support set、query、prototype 分别对应什么。
- 距离计算在哪里发生。
- 修改距离计算后，会不会影响训练、验证、测试。
- 如果新增 graph distance、kNN、Isomap、Floyd-Warshall 等方法，必须用通俗例子解释。
- 如果新增频域分支，必须说明它和原始像素域分支的关系。
- 如果修改距离计算或新增分支，必须保留原始方法作为 baseline 或提供开关。

通俗解释原则：

- prototype：某一类图片的平均代表。
- query：当前要判断类别的图片。
- support set：用来计算 prototype 的少量样本。
- embedding / feature：模型把图片转换成的一串数字特征。
- distance：query 和 prototype 之间的远近。
- 欧氏距离：两个点之间的直线距离。
- 图距离：先把相近样本连成图，再看两个点在图上走过去有多远。
- Isomap 思想：用 kNN 建图，再用图上的最短路径代替直线距离。
- 频域：把图片从像素变化转换成频率变化后看到的信息。
- 双分支：一条分支看原图或像素特征，另一条分支看频域特征，再把两边的信息合起来。

不要只讲数学公式，要说明它在代码里对应哪一步。

## 8. 风险控制

执行重要操作前必须提醒用户。

重要操作包括：

- 删除文件
- 覆盖文件
- 大规模重构代码
- 安装新依赖
- 修改训练入口
- 修改测试入口
- 修改数据读取逻辑
- 修改模型主干网络
- 修改实验输出路径
- 新建大规模实验分支
- 合并分支
- 使用 git reset --hard
- 使用 git push --force
- 删除分支

优先使用安全做法：

- 先查看 git status
- 先确认当前分支
- 先保留原方法
- 先加参数开关
- 先做小范围验证
- 不要同时在 Windows 和 AutoDL 上修改同一批代码文件
- 不要把多个大实验方向混在一个分支里

不要把以下内容提交到 Git：

- 数据集
- 模型权重
- 训练输出
- 缓存文件
- 大日志文件
- 临时文件
